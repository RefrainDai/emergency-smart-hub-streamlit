# 方法说明

## 1. 系统应用场景

应急智枢面向地震、洪涝、山火、危化事故等灾害响应场景。灾害现场会产生图像、视频、文本上报、求助信息和态势评估等多源任务请求，这些请求需要调用快速问答、目标识别、图像分析和报告生成等多模态智能服务。

在突发灾害中，请求负载具有波动强、业务异构和时效要求高等特点。如果只在请求到达后被动分配资源，容易造成等待服务队列增长、响应时延上升和资源利用失衡。因此，平台重点展示如何提前预测下一时隙资源缓存量，辅助应急中心进行资源预分配与调度决策。

## 2. 多模态任务服务负载

平台中的任务负载用于模拟不同多模态服务的请求到达情况，包括：

- 快速问答服务：用于低时延灾情问答和状态判断。
- 受灾目标识别：用于目标检测、受损区域识别和目标计数。
- 结构化报告生成：用于形成面向指挥中心的灾情摘要。
- 灾害图像深度分析：用于复杂图像理解与辅助研判。

这些负载被整理为按时间变化的请求序列，平台据此计算当前请求到达量、空闲算力资源、等待服务队列和即时需求缺口。

在演示模式中，平台新增多模态智能体任务接入层。该接入层不加载真实大模型权重，而是通过固定工具调用轨迹展示“任务规划、语义分割、灾害图像分析、结构化报告生成、结果回传”等处理过程如何转化为多类型任务服务负载。该负载随后进入既有的历史案例库、相似案例检索和 ICL 缓存决策流程。

## 3. 历史案例库构建

历史案例库由历史决策轨迹构成。每个案例包含：

- 任务类型；
- 近期请求窗口；
- 当前系统状态；
- 资源缓存决策；
- 请求保障率、资源冗余率和综合评价值等反馈结果。

案例库不是简单保存原始数据，而是将历史状态、动作和反馈评价整理为可检索的经验样本。高质量案例会为后续 ICL 少样本缓存决策提供上下文参考。

## 4. Top-K 相似案例检索

在当前时隙，平台根据当前任务状态从历史案例库中检索 Top-K 相似历史案例。相似度主要考虑：

- 任务类型是否一致或接近；
- 近期请求到达量的变化趋势；
- 空闲算力资源和等待服务队列是否相似；
- 历史案例的反馈评价质量。

相似案例检索的作用是让当前决策不只依赖当前瞬时状态，而能参考历史上相近场景下的资源缓存经验。

## 5. ICL 少样本缓存决策

ICL 少样本缓存决策将当前状态与 Top-K 相似历史案例组织为上下文输入，用于生成下一时隙资源缓存量。其核心思路是：

```text
当前状态 + Top-K 相似案例 + 上下文输入模板
→ ICL 少样本缓存决策
→ 下一时隙资源缓存量
```

平台默认可以在不配置外部模型接口的情况下完成本地演示；如需展示外部 ICL 推理过程，可在高级配置中接入外部模型接口。未配置或调用失败时，平台会回退到相似案例缓存预测。

## 6. 反馈评价与更新条件分析

资源缓存决策执行后，平台会根据当前时隙结果计算：

- 请求保障率：当前请求得到及时服务的比例；
- 资源冗余率：缓存资源中未被使用的比例；
- 综合评价值：综合考虑等待积压和资源冗余后的评价结果。

这些反馈可以用于分析新产生的案例是否具备进入案例库的价值。当前平台展示反馈评价和更新条件分析，不强行宣称已经完成自动在线写入案例库。

## 7. 对应急中心调度的辅助价值

该模块的输出不是直接控制真实硬件，而是形成面向应急中心的资源预分配建议。预测得到的资源缓存量可以辅助：

- 提前预热多模态模型服务实例；
- 预留 GPU 与显存资源；
- 调整不同任务服务的并发处理配额；
- 降低突发请求导致的等待队列增长；
- 在资源保障和资源冗余之间进行可解释权衡。

通过历史案例库、相似案例检索和 ICL 少样本缓存决策，平台为灾害响应多模态任务服务提供了可观察、可解释的资源缓存预测流程。
